# Technical Context

Architecture, design decisions, and known limitations of **SQL Schema Visualizer**. Read this before contributing.

## Overview

A 100% client-side single-page app. SQL DDL is parsed in the browser into an in-memory schema model, which is laid out with dagre and rendered as an interactive React Flow diagram. There is no backend, no network call, and no persistence beyond the browser's `localStorage`.

```
SQL text ──▶ parser ──▶ Schema model ──▶ React Flow nodes/edges ──▶ dagre layout ──▶ canvas
   ▲                         │
   └── localStorage          └── search / focus / comment-mode (Zustand store)
```

## Stack

| Concern | Choice | Why |
|---|---|---|
| Build | Vite 8 | fast HMR, simple SPA build |
| Language | TypeScript | parser correctness, node data typing |
| Routing | TanStack Router (file-based) | requested; one route today, room to grow (shareable URLs) |
| Canvas | React Flow (`@xyflow/react` v12) | battle-tested pan/zoom/drag, minimap, handles |
| Auto-layout | `@dagrejs/dagre` | directed-graph layout for FK relationships |
| State | Zustand | tiny, no boilerplate, selector subscriptions |
| Styling | Tailwind CSS v4 (`@tailwindcss/vite`) | utility-first, dark theme |
| Export | `html-to-image` | PNG of the React Flow viewport |

`src/routeTree.gen.ts` is generated by the TanStack Router Vite plugin and is git-ignored. `npm run build` (or `npx tsr generate`) regenerates it; `task routes` is a shortcut.

## Parser design (`src/lib/parser.ts`)

The parser is **hand-rolled**, not a SQL grammar library. Rationale:

- Multi-dialect tolerance beats strict correctness for a *visualization* tool. A real grammar for one dialect chokes on vendor clauses (`ENGINE=InnoDB`, `AUTOINCREMENT`, backticks); juggling four grammars is worse.
- Goal is a diagram, not validation: **skip-on-unknown, never throw**. A partially-understood dump still yields a useful picture, with `warnings[]` surfaced in the UI.

Pipeline:

1. **`stripComments`** — removes `--`, `#`, `/* */` while respecting string literals (used for the code-parsing pass only; comments are recovered separately).
2. **`splitStatements`** — splits on top-level `;`, tracking quote and parenthesis depth.
3. **`CREATE TABLE`** — extracts the table name (quotes/brackets/schema-prefix stripped via `cleanIdent`), splits the body on top-level commas, classifies each part as a column definition or a table constraint.
4. **Column parse** — name, type (leading token incl. parenthesised length, `UNSIGNED`, array `[]`), `NOT NULL`, `PRIMARY KEY`, `UNIQUE`, `DEFAULT`, inline `REFERENCES`.
5. **Constraints** — table-level `PRIMARY KEY (...)`, `UNIQUE (...)`, `FOREIGN KEY (...) REFERENCES ...`; plus `ALTER TABLE ... ADD FOREIGN KEY`.
6. **`associateComments`** — a second physical-line walk that re-attaches `--` / `#` comments to the right table/column.

### Comment association heuristic

`associateComments` walks lines, tracking the current table and last seen column:

- A comment on a `CREATE TABLE` header line → table comment.
- A comment on a column-definition line → that column.
- A **comment-only** line (continuation) → appended to the last seen column, else the table.

Consequence: a wrapped comment block whose continuation lands on the line *after* the header binds to the first column, not the table. The text is never lost, but its anchor can shift by one row. This is an accepted tradeoff for a forgiving line-based pass — a full positional parser would be far heavier for marginal gain.

## Rendering & interaction (`Canvas.tsx`)

- `baseNodes` / `baseEdges` are memoised from `schema`; `layout()` runs dagre and returns positioned nodes. The layout effect re-fits the view on schema/direction change.
- **Search** dims non-matching nodes (match = table name *or* any column name) and animates connected FK edges.
- **Focus** (click-to-navigate): a store `focus` object with an incrementing `nonce` retriggers an effect that `setCenter`s on the node and flags the focused column. `setCenter` runs inside `requestAnimationFrame` *after* the node update.
- **Gotcha (documented):** the React Flow `fitView` *prop* re-fits the viewport on every node change and silently cancels `setCenter`. It is intentionally **not** set as a prop — initial fit is done manually via the layout effect's rAF `fitView()` call. Do not re-add the `fitView` prop without rethinking focus navigation.
- **Hover popovers** escape the node bounds; `.react-flow__node:hover { z-index: 1000 }` (in `index.css`) lifts the hovered node above siblings so popovers aren't occluded.
- **Navigation model:** `panOnScroll` + `panOnScrollMode=Free` + `zoomOnScroll=false` + `zoomActivationKeyCode=['Meta','Control']` = Figma/trackpad feel. Magic Mouse has no pinch hardware; zoom there is ⌘/Ctrl+scroll, double-click, or the controls.

## State (`store.ts`)

Single Zustand store. Persisted to `localStorage`: `dbviz.sql` (last SQL), `dbviz.comments` (`off` | `inline` | `hover`, with legacy `0`/`1` migration), `dbviz.theme`, `dbviz.groups.v1` (versioned: `{groups, activeGroup}`). `search`, `focus`, `direction`, `history` (FK back-stack) are session-only.

## FK navigation (`Canvas.tsx` + `store.ts`)

- Click the `↗` glyph on any FK column → `focusTable(targetTable, targetColumn)` pushes the current focus onto a 50-entry `history` stack and animates `setCenter` on the target.
- Click the relationship edge → ping-pongs between source and target.
- `back()` action pops the stack; bound to the toolbar `← Back` button and the `⌥/Alt + ←` / `⌘/Ctrl + [` keyboard shortcuts (capture-phase listener so React Flow can't intercept first). Backspace deliberately untouched — that's React Flow's "delete selected node" default.

## Table groups (`store.ts` + `Canvas.tsx` + `GroupsPanel`)

- Two sources, same UI: user-managed `groups` (CRUD via store actions, persisted) and `schema.groupAnnotations` (read-only, derived from `-- @group: name` line comments above each `CREATE TABLE`; a table can be in multiple groups).
- `computeVisibleSet(schema, groups, activeGroup)` returns the filtered set; `edgeIsVisible(set, src, tgt)` is the matching edge predicate. Cross-boundary FK edges drop while filtered. Both helpers live in `packages/react/src/groups.ts` and are exported for custom canvases.
- Right-click context menu (`GroupsContextMenu`) acts on the single right-clicked node — unless that node is part of a multi-selection (Shift- / ⌘-clicked, or rubber-band selected), in which case the menu's add/remove buttons act on the **whole selection** with bulk semantics ("Add 3 (1 already in)").
- Auto-clear: if a group is the active filter and its members all vanish from the SQL on reparse, `activeGroup` resets to null to avoid a blank canvas.

## Share URL codec (`packages/core/src/share.ts`)

`#s=<token>` carries the compressed SQL. `&g=<token>` (optional, additive) carries `{g: groups, a: activeGroup}` JSON via the same deflate-raw + base64url pipeline. Pre-groups viewers ignore `&g=`; new viewers without groups omit it. See `buildShareUrl(sql, {groups, activeGroup})` in the React package.

## Backup (`packages/react/src/backup.ts`)

JSON snapshot of `{sql, groups, activeGroup, commentMode, theme}` with a `BACKUP_VERSION` field. `validateBackup` tolerates garbage and rejects unknown majors with a clear error. `applyBackup` drives store actions in safe order (SQL → wipe-then-rebuild groups → activeGroup → preference toggles). `<ExportBackupButton>` / `<ImportBackupButton>` are thin React wrappers; the helpers are usable headless.

## Help (`packages/react/src/help.ts` + `Help.tsx`)

`defaultHelpEntries` is a typed `HelpEntry[]` (id / section / title / body / optional shortcut chip + keywords). `matchHelpEntry` is a token-substring match across title + body + keywords. `<HelpButton>` (and global `?` shortcut) opens `<HelpModal>`, which groups results by section; both components accept an `entries` prop so consumers can extend (`[...defaultHelpEntries, ...mine]`) or fully replace.

## Known limitations

- Not a real SQL grammar — exotic DDL (generated columns, deeply nested parens, CTE-in-DDL, partition clauses) parses loosely.
- Comment continuation anchoring can shift by one row (see heuristic above).
- Hover popover anchors right (`left-full`); near the right viewport edge it may extend off-screen — pan to read.
- No live database introspection (by design — client-only, privacy-first).
- Layout is dagre-only; very wide schemas produce long horizontal spans (toggle vertical or pan).
